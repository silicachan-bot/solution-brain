以下是 B 站某视频评论区里的“评论对”。每一项都包含：
- 上文评论
- 针对它的回复评论

请只从“回复评论”里发现值得收录的语言模式。

把“是否收录”当作严格筛选题，而不是发散找亮点。只有同时满足多数批准条件的回复，才值得提成模式。

批准标准：
- 它本身像一句“回复别人时会说的话”，而不是围观、分析或补充信息
- 离开当前视频和具体话题后，仍然大致成立，能迁移到别的对话场景
- 句子有比较稳定的固定构式，或有明确可替换槽位，不是随口说一句什么都能套
- 语气鲜明，像一个完整的回话动作，带有惊讶、调侃、反驳、接梗、追问、抱怨等互动性
- 即使只出现一次，也能看出它不是普通白话，而是有明显“句式感”的表达
- 压缩成模板后，例句仍然和模板严格对应，不会出现“模板很抽象、例句其实对不上”的情况
- 模板应尽量保留原句骨架、口语感、语气词和标点，不要把一句有风格的话改写成普通书面句
- 只有替换位非常明确时才使用 `[A]` `[B]`；不要为了做成模板而强行造槽位
- 很短的回复也可以保留，但前提是它的结构本身有辨识度，不是普通附和词
- 角色扮演式开场、主播腔开场、整活宣告式开场，如果结构稳定且可替换，也可以保留
- 惊讶、反问、接梗类短句，如果一句话本身就有明显固定说法，也可以保留

不要收录这类模式：
- 太普通的感叹、附和、吐槽，换成别的词也没有稳定句式价值
- 严重依赖视频内容、角色设定、评论区上下文或当前对象，脱离原场景就不自然
- 更像围观评论，不像拿来回别人时会说的话
- 事实讨论、剧情分析、信息补充、长解释、复述观点
- 对某个具体事件做总结、安慰、解释后，硬抽出来的宽泛句子
- 把某人的成功、现象或结果归因到身份、背景、资源的“揭底句”
- 给当前对象提建议、推荐替代去处、劝人降低预期的场景化回复
- 纯评论区链路话术，例如“回复 @某人 : …”
- 纯评论区互动口号、抢前排话术、刷存在感话术
- 只有一句有意思，但没有稳定构式、不能自然复用的单条评论
- 模板里虽然能塞 `[A]`，但实际只是把原句硬拆成槽位
- 完整的评价句、鸡汤句、结论句，看起来工整但本质上不是高复用回复模板
- 把原句改写得比原评论更书面、更泛化，导致味道和结构都变了的模板

宁缺毋滥。一批最多输出 3 个高质量模式；如果没有合适的，返回空数组 []。

对每个发现的模式，输出 JSON 数组，每个元素包含：
{
  "template": "含 [A] [B] 占位符的模板句",
  "examples": ["2-5个真实回复例句"],
  "description": "模式描述：是什么、什么时候能用于回复、传达什么感觉"
}

要求：
- `examples` 必须来自回复评论，不要引用上文评论
- 只有在回复里真的体现出稳定句式时才输出
- 如果只是单条评论有趣、但不像稳定可复用模式，不要输出
- 单例也可以输出，但前提是这句话本身有明显固定构式、梗感或可替换槽位
- 尽量把模式抽象成能复用的模板，而不是原句照抄
- 不要因为一句话“看起来能填 [A]”就硬抽模板；离开原评论话题就不自然的句子不要输出
- 优先保留“短、硬、像一句回话”的句子；谨慎保留宽泛安慰句、概括句、评价句
- 不要擅自把原句“规范化”“补全”或“改写”为另一种句式；模板首先要像原话本身
- 如果原句的风格点就在固定说法本身，就保留固定说法，不要再拆成槽位
- 如果多个候选里有“更像完整回话动作”的句子，优先保留它，不要把名额给宽泛评价句
- 如果一句话的核心价值主要来自“具体名词”而不是“句式结构”，不要提取
- 如果一句话更像信息判断、立场分析、对象推荐，而不是互动回复，不要提取

只输出 JSON，不要其他文字。
